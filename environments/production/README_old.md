
# Retail Product Catalog - GitOps Infrastructure Manifests (Repo-3)

This repository serves as the single source of truth for the desired state of the **Production 3-Tier Retail Application** infrastructure inside our Amazon EKS cluster. 

**ArgoCD** continuously monitors this repository. Any modification pushed to the `main` branch is automatically synchronized into the live cluster environment, eliminating configuration drift and manual tracking.

---

## 📂 GitOps Directory Blueprint

```text
retail-infra-gitops/
└── environments/
    └── production/
        ├── argocd-app.yaml        # Root Bootstrapper Application Manifest
        ├── secrets-eso.yaml       # AWS Secrets Manager Integration via ESO
        ├── network-policies.yaml  # Strict 3-Tier L4 Firewalls (Isolation)
        ├── backend-deploy.yaml    # API Cluster Deployments & Internal Services
        └── frontend-deploy.yaml   # Web Frontend NodePorts & AWS ALB Ingress

```

## 🚀 Step-by-Step Production Deployment Guide

Follow these sequential steps to configure, link, and bootstrap your declarative delivery platform.

### Step 1: Update Application Container Images

Before synchronization can occur, you must update the Kubernetes manifests to point to the secure Amazon ECR container paths generated by your CI build pipeline (**Repo-2**).

1. Open `environments/production/backend-deploy.yaml`. Locate the `image:` attribute on line 20 and input your unique ECR repository location:
```yaml
image: [123456789012.dkr.ecr.us-east-1.amazonaws.com/retail-backend:latest](https://123456789012.dkr.ecr.us-east-1.amazonaws.com/retail-backend:latest)

```


2. Open `environments/production/frontend-deploy.yaml`. Locate the `image:` attribute on line 18 and mirror the configuration for your web layer asset:
```yaml
image: [123456789012.dkr.ecr.us-east-1.amazonaws.com/retail-frontend:latest](https://123456789012.dkr.ecr.us-east-1.amazonaws.com/retail-frontend:latest)

```



---

### Step 2: Establish Your Live AWS Secrets Target

The **External Secrets Operator (ESO)** relies on an explicit secret object within AWS Secrets Manager to pull runtime credentials into the cluster keylessly.

1. Navigate to the **AWS Management Console** > **Secrets Manager**.
2. Click **Store a new secret** and configure a **Key/value** secret pattern.
3. Define the exact database credential pairing paths matching your Terraform RDS state outputs:
* **Key:** `username` | **Value:** `dbadmin`
* **Key:** `password` | **Value:** `YourSecurePasswordValueHere`


4. Name the secret location exactly as follows: **`production/rds/credentials`**.
5. Save the secret using standard cloud encryption configurations.

---

### Step 3: Configure the Root Bootstrapper Mapping

ArgoCD uses the "App-of-Apps" pattern. You feed it a single configuration file that points to your public or private Git repository, and it automatically installs everything else found inside that directory.

1. Open `environments/production/argocd-app.yaml`.
2. Locate the `repoURL:` key on line 11 and change it to your exact GitHub link:
```yaml
repoURL: '[https://github.com/Chinthaparthy-UmasankarReddy/retail-infra-gitops.git](https://github.com/Chinthaparthy-UmasankarReddy/retail-infra-gitops.git)'

```



---

### Step 4: Ship and Link the Manifest Infrastructure

Push all of your configured declarations to your remote Git platform:

```bash
# 1. Access your local working root directory
cd D:\Projects\3-Tier_Microservices_Application\retail-infra-gitops

# 2. Stage and commit tracked alterations
git add .
git commit -m "config: personalize production image targets and repo locations"

# 3. Synchronize your local trunk upstream
git branch -M main
git push -u origin main

```

---

### Step 5: Bootstrap ArgoCD on the Cluster

With your code securely pushed upstream, execute the initial connection phase to bind your EKS context to your deployment definitions.

1. Point your local command terminal to your managed AWS compute engine:
```bash
aws eks update-kubeconfig --region us-east-1 --name production-eks-cluster

```


2. Inject the root declarative deployment token manually **exactly once**:
```bash
kubectl apply -f environments/production/argocd-app.yaml

```

## 🔍 Platform Verification & Health Matrix

Once bootstrap components are fed into the system, monitor the convergence loops to verify system initialization:

### 1. Track Cluster Synchronization Progress

Log into your web-facing **ArgoCD Dashboard** or run command queries directly from your desktop window to watch the platform spin up dependencies:

```bash
kubectl get apps -n argocd

```

*Expected Output status metrics:* `Synced` and `Healthy`.

### 2. Verify Dynamic Secret Interception

Confirm that the External Secrets Operator has correctly verified your cloud infrastructure configurations and mapped your database keys into standard Kubernetes namespaces:

```bash
kubectl get externalsecret rds-db-secret -n production
kubectl describe secret app-db-credentials -n production

```

### 3. Check Real-Time Network Routing

Confirm that your public **AWS Application Load Balancer (ALB)** has finished provisioning external ingress endpoints to handle web traffic boundaries:

```bash
kubectl get ingress retail-application-ingress -n production

```

Copy the resulting long address (`*.amazonaws.com`) directly into your web browser to access your live, firewalled, production-monitored 3-tier product catalog platform!
